# CrediFlow 授信风控管线全流程说明

本文档详细描述了 CrediFlow 系统中，用户从发起授信申请到最终额度生效（或拒绝）的端到端风控全流程。流程设计以“Java 确定性分流为主，AI Agent 辅助洞察为辅”为核心原则。

---

## 一、准入前置与发起

1. **发起条件：KYC 与主卡校验**
   用户在 APP 点击「开通授信额度」前，必须通过 `user-service` 完成 KYC 实名实人核验，并至少绑定一张主银行卡。
   - 若未完成，前置网关直接拦截。
2. **初始化授信申请单**
   风控服务（`credit-risk-service`）收到请求后，生成一条初始状态为 `PENDING_HARD_RULES` 的授信申请记录（申请额度系统默认初始占位，如 5000 元）。

---

## 二、第一阶段：Java 确定性风控评估

本阶段完全由 Java 系统通过规则和公式计算，确保底层风控的绝对客观与可审计性。

### 1. 基础硬规则管线拦截 (Hard Rules)
申请单首先进入 `HardRuleEngine`。
- **校验项**：
  - **`BR_001`**: 命中外部多头借贷黑名单 (由三方征信数据源拉取)
  - **`BR_002`**: 历史存在未结清的逾期贷款 (账单表关联核实)
  - **`BR_003`**: 账户当前处于高危风险冻结状态或非正常状态
- **处理结果**：
  - **不通过**：直接将申请单状态置为 `REJECTED`，并记录拒绝审计代码（如 `BR_001: 命中外部黑名单`）。
  - **通过**：流转至 `PENDING_SCORING`。

### 2. 四维加权模型评分 (Scoring Engine)
核心风控评分公式（Risk Scoring Formula）由 S1 至 S4 四个维度的因子加权计算得出：

**评分公式：**
> **$$ TotalScore = \sum_{i=1}^{4} (W_i \times S_i) $$**

**具体权重分配：**
- **$S_1$ (基础信用分, 权重 $W_1 = 0.2$)**: 基于用户年龄、学历、职业稳定性的静态评估。
- **$S_2$ (履约能力分, 权重 $W_2 = 0.4$)**: 基于用户的银行流水账单、公积金缴纳记录评估的还款能力（核心占比最大）。
- **$S_3$ (多头借贷分, 权重 $W_3 = 0.2$)**: 跨平台借贷申请频次，频次越高该分数越低。
- **$S_4$ (行为偏好分, 权重 $W_4 = 0.2$)**: APP 内操作轨迹（如连夜多次尝试申请、设备频繁切换等高危特征扣分）。

**具体计算示例：**
$$ TotalScore = 0.2 \times S_1 + 0.4 \times S_2 + 0.2 \times S_3 + 0.2 \times S_4 $$

**等级阈值划分 (Risk Level Classification)：**
| 总分区间 | 模型风险等级 (`model_risk_level`) | 业务含义 |
| :--- | :--- | :--- |
| $$ TotalScore \ge 85 $$ | **LOW (低风险)** | 极度优质客户 |
| $$ 60 \le TotalScore < 85 $$ | **MEDIUM (中风险)** | 普通大众客户 |
| $$ TotalScore < 60 $$ | **HIGH (高风险)** | 高危高敏客户 |

---

## 三、第二阶段：确定性矩阵路由与活体校验

获得模型风险等级后，申请单进入 `PENDING_ROUTING` 阶段，Java 核心依据以下矩阵进行绝对确定性的路由：

| `model_risk_level` | 二次人脸要求 | 最终流向 |
| :--- | :--- | :--- |
| **LOW (低风险)** | **免人脸** | **自动通过 (`APPROVED`)** |
| **MEDIUM (中风险)** | **强制触发** | 人脸通过 → **自动通过 (`APPROVED`)** <br> 人脸失败 → **拒绝 (`REJECTED`)** |
| **HIGH (高风险)** | **强制触发** | 人脸通过 → **转人工审核 (`PENDING_MANUAL_REVIEW`)** <br> 人脸失败 → **拒绝 (`REJECTED`)** |

*注：此阶段触发二次人脸时，申请状态进入 `PENDING_SECONDARY_FACE` 等待活体识别服务的异步回调。*

---

## 四、第三阶段：人工与 Agent 协同审核

只有**高风险**（且通过二次人脸）的用户，或者被特殊事件降级的用户才会进入人工审核阶段。

### Agent 辅助分析机制（转人工三件套）
AI Agent 严禁直接下发“拒绝”或“放行”指令干预主链路。当申请落入人工队列 (`cf_credit_review_queue`) 后：
1. Agent 会在后台异步拉取该用户的所有特征及评分画像进行深度分析。
2. **输出审核三件套**展示给后台管理员：
   - **具体风险明细**：自然语言提炼出各项异常（如“近期多头借贷频繁”、“常用设备发生跨省登录异常”）。
   - **双维量化概率**：
     - **违约概率 (Default Probability)**: `1.0 - (TotalScore / 100.0)`
     - **欺诈概率 (Fraud Probability)**: 基于知识图谱与黑产设备库的交叉匹配，高风险基础为 `35%`，其余为 `10%`。
   - **操作建议**：必须输出四大枚举建议之一（建议放行 / 建议降额 / 建议拒绝 / 建议限制期数）。
3. 最终由人类审核员参考 AI 建议，在管理台执行业务判定（`APPROVED` 或 `REJECTED`）。

---

## 五、特殊风控补丁：对话意图安全网

为了防止用户在主流程之外暴露出高危意图（如在客服系统试图套现），加入了实时的意图风险监控机制。

- **触发场景**：用户在 APP 端与 AI 助理进行自然语言聊天时。
- **意图捕获**：大模型识别用户话术中包含 **逃废债**、**不想还款**、**套现诱导** 等关键字（如“没钱”、“不想还”、“投诉”）。
- **降级预警 (`Chat Risk Escalation`)**：
  - Agent 不直接截断主链路，而是将违规意图（`INTENT_EVASION`）及**导致告警的完整聊天记录原话** 发往后台 `cf_credit_review_queue`。
  - 管理员可在后台根据用户聊天记录，人工决定是否要冻结其已有额度。

## 六、终态处理与额度计算

### 1. 审批通过 (`APPROVED`) 与 额度生成
一旦流转到 `APPROVED`，系统将基于风险总分使用线性插值算法来赋予用户最终的循环授信额度。

**额度计算公式 (Quota Calculation Formula):**
> **$$ UserQuota = MinQuota + \frac{clamp(TotalScore, 60, 100) - 60}{100 - 60} \times (MaxQuota - MinQuota) $$**

**公式变量说明：**
- **$MinQuota$**: 系统准入的最低保底额度（默认 `1000.00` 元）。
- **$MaxQuota$**: 系统允许的最高授信天花板（默认 `50000.00` 元）。
- **$TotalScore$**: 第一阶段计算出的四维加权总分。
- **$clamp(x, min, max)$**: 钳位函数，确保用于计算的分数在有效区间内，由于 60 分以下高风险转人工，自然进来的基础分均视为 $\ge 60$。

**计算逻辑**：
将 60~100 分的 40 分差等比例映射到 1000~50000 元的资金区间。
例如：若某用户 $TotalScore = 80$，则额度为：
$1000 + (80-60)/40 \times 49000 = 1000 + 0.5 \times 49000 = 25500.00$ 元。

额度计算完毕后，将持久化写入 `cf_user_credit_quota` 循环账户体系。用户需完成《授信电子合同》(`CREDIT_CONTRACT`) 的签署（状态变为 `SIGNED`）后，底层可用额度方才正式生效，进而可发起借款申请。

### 2. 审批拒绝 (`REJECTED`) 与 拒绝洞察
- 申请被驳回时，风控系统调用 Agent `credit_rejection_insight` 生成脱敏后的拒绝原因。
- **`userSafeInsight`**: 面向 C 端用户的安抚文案（如：“由于您近期的风险评分波动，暂时无法为您提供额度，建议保持良好信用记录后三个月再试”）。
- **`adminInsight`**: 面向后台管理员的真实拦截原因（如：“命中 BR_001 多头借贷黑名单”）。这确保了前端业务不直接向黑产暴露底层规则码。
