# 架构哲学指南：为什么我们放弃了应用层认证注解？

在 CrediFlow 向基于微服务网关和零信任架构演进的过程中，我们做出了一个重要决定：**彻底废弃** `@IgnoreAuth`（标记忽略登录校验）和 `@Inner`（标记仅限内网访问）等原本散布在各个 Controller 方法上的应用层安全注解。

这并非因为注解不好用，而是基于现代分布式架构对“安全性、确定性和性能”的三重考量。本文将阐明这两种安全治理哲学之间的差异与权衡。

## 两种安全哲学的对撞

### 哲学 A：基于代码的声明式控制 (AOP 注解)
典型的例子是我们在 Spring 中常用的 `@PreAuthorize` 或自定义的 `@Inner`。
- **优点**：代码可读性极强。开发者在看一个 Controller 的时候，能直观地知道这个接口的安全约束。
- **痛点**：
  1. **防线过深**：恶意请求需要穿透 Nginx、网关、甚至是微服务容器的负载均衡，一直打到具体的 Controller 方法上，在反序列化 JSON 的前一刻才被拦截。这意味着攻击者可以用极低的成本消耗你后端系统的计算资源。
  2. **默认放行（Default Allow）**：如果开发者在新写一个内部接口时**忘记**加 `@Inner`，或者切面（AOP）配置失效，这个接口会毫无防备地暴露给全网。
  3. **多语言灾难**：在 Go 或 Python 写的节点中，你需要手写类似的拦截器，无法做到跨语言的统一。

### 哲学 B：基于基础设施的声明式控制 (网关与路径约定)
这就是我们目前所采用的方案：利用 APISIX 网关作为前置哨剑，以及利用 `/api/internal/` 的统一前缀规范来界定内网。
- **优点**：
  1. **默认拒绝（Default Deny）**：任何尝试访问包含 `/internal/` 的路径的外部请求，会被网关在第一层直接掐断返回 403。即便你后端什么保护都没做，依然绝对安全。
  2. **防线前置**：无需唤醒 JVM，无效流量在网关层就已被清洗。
  3. **集中管理与跨语言**：只需要在网关配一次，不管是 Java、Go 还是 Python 写的接口，一视同仁。
- **痛点**：
  1. 代码可读性下降。在 Java 代码里，你只能看到一个 `@RequestMapping("/api/internal/user/info")`，而不是一个直观的 `@Inner` 注解。

## 我们的权衡与决议

在经历了一段时间的混合使用后，我们发现：**在微服务架构下，保留没有实际拦截作用的“视觉装饰”注解（如 `@Inner`）是最危险的。** 因为它会给开发者带来一种“**伪安全感**”——他们会以为只要加上注解就万无一失，从而可能忽视了将其放在 `/api/internal/` 路径下的硬性要求。

因此，我们做出了彻底的“断舍离”：

1. **废弃 `@IgnoreAuth`**：网关层的 `jwt-auth` 插件现在是判定接口是否需要验证 Token 的**唯一权威**。需要免登录的接口必须加入 APISIX 的白名单配置。业务层不必再通过注解自行标记，也没有权力私自放行未经网关认可的流量。
2. **废弃 `@Inner`**：内部互调的安全完全由 `InternalAuthFilter` 和 HMAC 签名（`X-Internal-Sign`）把控。只有路径匹配 `/api/internal/**` 且具备正确内部签名的请求才被认为是合法的内部调用。

## 给开发者的指引

**问：我需要写一个不需要用户登录就能访问的接口，怎么办？**
答：直接写你的 Controller，无需任何注解。然后，去 `infra/apisix/` （或对应网关配置源）中，将这个接口的路径加到 `jwt-auth` 插件的白名单（或单独剥离出一条不需要开启 JWT 插件的路由）中。

**问：我要写一个只能由系统内其他微服务调用的接口，怎么办？**
答：将你的 Controller 映射路径以 `/api/internal/` 开头（例如：`/api/internal/fund/callback`）。只要做到这一点，它自然会被 `InternalAuthFilter` 拦截并要求严苛的内网签名，外网用户绝对无法访问。
