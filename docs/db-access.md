# 数据库访问规范 (10.4)

## 规范说明

在微服务架构中，为了防止数据耦合与跨模块写冲突，**禁止调度器（`batch-service`）直连 MySQL 业务库进行写入操作**。

### 具体要求

1. **写操作必须通过 HTTP/RPC**
   调度器所有的写操作（例如触发代扣、生成罚息记录） MUST 通过请求对应微服务暴露的内部 API（如 `/api/internal/...`）来完成。
   微服务内部实现幂等性与写入的本地事务。

2. **读操作的安全约束**
   如果调度器需要进行海量数据的扫描（例如找出所有未还款的计划），并且通过 HTTP 拉取分页数据成本过高：
   - **允许** 调度器通过直连 MySQL 库进行只读查询（Read-Only）。
   - **必须** 使用配置有只读权限（Read-Only User）的数据库账号。
   - **推荐** 查询从库（Slave/RO Instance）以避免对主库造成 CPU 与 I/O 压力。

### 当前配置与方案
在目前的 `batch-service` 实现中，完全采用**纯 API 驱动**（向 `repayment-service` 和 `post-loan-service` 发起 HTTP 调度调用）。
若后期数据量膨胀需直连扫描，请按上述要求配置 `RO_DB_DSN` 并在 `go.mod` 中引入 `gorm`。
