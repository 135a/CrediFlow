## Why

当前放款、还款链路对资金方的对接散落在 Java 业务微服务（`fund-flow`、`repayment` 等）之中，存在三个问题：一是 Java 业务层直连资金方使得加签、加密、超时、熔断和回调验签逻辑分散，难以统一治理；二是资金方接口地址、AppKey/AppSecret 和加密密钥可能散落在 Java 工程的配置里，无法在多环境、多家资金方之间安全切换；三是放款/还款资金流路径没有在规格层明确「资金方对公账户 ↔ 用户储蓄卡」是唯一资金通路，存在合规与审计风险。本次变更将所有资金方对接收敛到 Go 服务层（统一网关），并明确角色边界（资金方为唯一持牌出/收资方、支付代扣通道仅为转账工具），让 Java 业务侧只感知内部「Go 资金接入网关」契约。

## What Changes

- 新增 Go 侧统一「资金方接入网关」（fund-channel-gateway，归属 `go/` 工程），作为本系统对所有资金方放款、还款、代扣、查询、回调接口的**唯一对外出口**；Java 业务微服务 MUST NOT 直连任何资金方接口。
- **BREAKING**：`microservice-fund-flow` 不再要求由 Java 通过「第三方放款接口」直接出金；改为通过 Java→Go 内网调用，由 Go 完成资金方加签/加密/重试/熔断/回调验签后再回传业务结果，原 `第三方支付对接预留` 与 `放款与出金流水对接` 相关场景语义更新。
- **BREAKING**：`microservice-repayment` 中主动还款的「调用第三方网关进行收银」改为由 Java 组装还款订单参数后下发给 Go 服务，由 Go 调用资金方还款/代扣 API；`scheduler-go` 中的自动代扣调度不再调用 Java 还款服务的 `内部扣款接口`，改为直接调用 Go 资金网关的代扣接口。
- 资金方接口地址、AppKey、AppSecret、加签/加解密密钥、请求超时、重试与熔断阈值全部托管在 **Nacos**，支持开发/测试/生产一键切换以及多家资金方动态启停与无缝替换；代码中 MUST NOT 出现硬编码的资金方密钥或地址。
- 资金通路在规格层固化为：放款由资金方对公账户直发用户绑定储蓄卡；还款由用户绑定储蓄卡原路回流资金方对公账户。系统服务器与平台账户 MUST NOT 出现资金过路；支付代扣通道在规格中明确为「资金转账工具」而非「资金方」。
- 在通信安全层固化要求：协议 HTTPS、请求/响应 JSON、请求加签 + 响应验签、敏感字段 AES/RSA 加密、请求级幂等键防重放款/重复代扣、超时重试 + 熔断降级；并要求资金方异步回调走 Go 网关统一回调接口，验签通过后再桥接到 Java 业务侧（订单状态、分期账单、授信额度恢复、用户画像履约标签）。
- 明确放款/还款端到端流程的服务接力顺序（用户 → Java 业务校验 → Python 风控 → Java → Go → 资金方 → 异步回调 → Go → Java → 履约标签/画像），并要求 Java↔Go 内网调用复用 `internal-api-security` 中的内网签名约束。

## Capabilities

### New Capabilities

- `fund-provider-go-gateway`：Go 侧统一资金方接入网关能力。覆盖 ①资金方接口契约抽象（放款、还款、代扣、查询、异步回调）；②基于 Nacos 的资金方多租户配置与多环境切换；③HTTPS + JSON + 加签验签 + AES/RSA 字段加密；④请求级幂等键、超时重试与熔断降级；⑤异步回调接收、验签与向 Java 业务侧的桥接事件/接口；⑥Java↔Go 内网调用契约（订单号、用户 ID、金额、分期、绑卡信息等业务参数到资金方报文的映射）。

### Modified Capabilities

- `microservice-fund-flow`：`第三方支付对接预留`、`放款与出金流水对接` 的实施路径从「Java 直接调用第三方放款/支付接口」修改为「Java 调用内部 Go 资金网关、由 Go 调用资金方」；放款回调由 Go 验签后回传，资金流水入账以「资金方→用户卡」直发口径登记，平台账户 MUST NOT 作为资金过路账户。
- `microservice-repayment`：`主动还款与幂等`、`自动生成还款计划与主动还款` 中的「第三方网关收银」语义更新为「调用 Go 资金网关」；主动还款与定时代扣两种触发都通过 Go 网关执行；扣款资金回流路径在规格中明确为「用户绑定储蓄卡 → 资金方对公账户」。
- `microservice-loan-application`：`借款通过后的异步事件分发` 增加放款执行边界，明确借款通过事件最终触发 Go 网关放款链路（Java 仅做参数下发、不直连资金方）。
- `microservice-post-loan`：增补放款/还款回调结果驱动履约/画像与逾期更新的语义入口（来自 Go 网关回调结果）。
- `scheduler-go`：`每日自动代扣还款调度` 的目标接口从 Java 还款服务的内部扣款接口修改为 Go 资金网关的代扣接口（由 Go 统一加签、加密、熔断与回调编排）。
- `internal-api-security`：明确 Java→Go 资金网关、Go→Java 业务回调桥接接口同样受内网签名约束，并增加资金类高敏接口的最小化字段与脱敏要求。

## Impact

- **代码**：
  - 新增 `go/fund-channel-gateway`（或在现有 Go 工程中新增模块），承载资金方 SDK 封装、加签/加密、重试熔断、回调入口与桥接出口。
  - `java/fund-flow-service`、`java/repayment-service`：移除/废弃直连第三方资金接口的客户端，替换为对 Go 资金网关的内部 Feign 调用；保留账务、流水、对账、还款计划等核心业务。
  - `go/scheduler-service`：自动代扣触发改为调用 Go 资金网关。
  - `java/loan-application-service`、`java/post-loan-service`：感知放款/还款结果异步事件（来自 Go 回调桥接）。
- **配置**：Nacos 增加 `fund-provider.*` 命名空间（按 dev/test/prod 与多家资金方区分 group/dataId）。Java 工程移除资金方硬编码配置，仅保留 Go 网关内网地址。
- **依赖与协议**：标准 HTTPS、JSON；引入或复用统一签名/AES/RSA、幂等键、熔断器（Go 侧 sentinel-golang 或等价组件）。
- **MQ/事件**：放款/还款结果回调改由 Go 触发后桥接为内部事件（如 `FUND_DISBURSED_EVENT`、`REPAYMENT_SETTLED_EVENT`），Java 业务消费方逻辑不变但来源调整。
- **数据**：资金流水、对账明细维持现有表结构，新增「资金方标识 + 资金方流水号 + 加密载荷摘要」等字段以便回溯。
- **不在范围**：资金方真实接入（具体厂商对接细节）、前端改造、Python 风控算法变更；只锁定平台侧角色边界、Go 网关契约、Nacos 配置规范与放款/还款端到端流程。
