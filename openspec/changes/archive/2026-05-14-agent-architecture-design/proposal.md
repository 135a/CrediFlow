## Why
当前系统虽然拆分了部分微服务，但在信贷全生命周期中，业务边界的定义依然需要更高的抽象和内聚。为了明确系统的领域划分、降低耦合度，并充分发挥 Java 和 Python 在不同计算密集度或业务流转场景下的优势，我们需要根据《信贷项目智能体设计与分工文档.md》引入“智能体（Agent）”化的架构思维。这能够为系统后续扩展（如合规审查智能体、报表智能体等）打下坚实的基座，并使得各业务域具备极强的独立开发和本地测试能力。

## What Changes
- 确立系统的 **5 大核心智能体（Agent）架构**：用户服务 Agent、反欺诈与风控 Agent、额度定价 Agent、合约流程 Agent、贷后管理 Agent。
- 明确这些 Agent 的核心职责边界、所选用的底层技术栈（Java / Python）及其在信贷上下游的流转调用关系。
- 制定 Agent 间通过网关协同（或基于内部消息流转）的交互闭环方案。

## Capabilities

### New Capabilities
- `agent-architecture-overview`: 新增架构级规范文档。详细定义五大核心 Agent 的定位、职责划分、技术栈约束（Java Spring Cloud 侧重业务流转，Python 侧重规则引擎与分析定价），以及信贷全流程的串联协同机制。

## Impact
- **架构设计**: 为所有的微服务或模块提供了一个高层的“智能体领域”映射关系，重塑系统的认知模型。
- **开发测试**: 强调了各智能体的单一职责和独立可测性（本地可 Mock 测试），解除了多团队联合开发的阻塞点。
- **扩展性**: 后续若需接入 AI 或外部数据源，可以精准定位到具体的业务 Agent 中，而不影响其他环节。
