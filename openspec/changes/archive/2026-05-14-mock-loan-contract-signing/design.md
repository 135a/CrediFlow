## Context
当前 CrediFlow 系统在信贷全链路推进时，由于真实借款合同生成依赖于复杂的 PDF 模板渲染、第三方电子签章接口网关调用以及 OSS 云存储，导致系统无法低成本进行本地的前后端状态联调与测试。为此，我们设计一套“模拟合同流转”方案，脱离所有外部中间件与三方服务依赖，仅利用本地状态或数据库，全真模拟从“合同下发”到“用户签署”全业务逻辑。

## Goals / Non-Goals

**Goals:**
- 提供“生成合同”、“查询合同”、“签署合同”三个核心接口的 API 支撑。
- 合同的数据交互形式为纯 JSON，需包含核心字段：合同编号、用户信息、借款信息、年化利率及流转状态。
- 实现 `INIT` (待签署) -> `SIGNED` (已签署) 的状态机驱动，确保下游放款流程能被顺利触发。
- 在代码中以标准的 `TODO` 格式留下未来真实第三方组件接入的挂载点。

**Non-Goals:**
- **不涉及** PDF 文件的生成与网络传输。
- **不涉及** 调用真实的 E-sign 或 CA 认证机构。
- **不涉及** 开发专用的配置后台去管理多版本合同模板。

## Decisions

**Decision 1: 合同对象的持久化与展示形态**
- **方案选择**：虽需求提及“完全内存模拟”，但为了降低开发心智负担以及方便后续重启依然保持状态联调，我们将直接复用已有的 MybatisPlus 实体 `LoanContract` 及 `cf_loan_contract` 表，不新建专门的 `HashMap` 内存缓存。合同的生成（`INIT`）和签署（更新为 `SIGNED`）操作均直接操作该表。返回给前端的“模拟合同”就是对这行数据的 JSON 序列化输出。
- **Rationale**：复用现成表结构能最快达到“模拟返回”且状态留痕的目的，且后续接入真实 PDF 时，表结构本身也不需伤筋动骨的改变。

**Decision 2: 接口定义与 TODO 埋点**
- **方案选择**：在 `LoanContractController` 及 `LoanContractServiceImpl` 中补充或优化 `generate`、`query`、`sign` 的核心逻辑。
  - **TODO 1**: 真实接入时，需要在 `generate` 后异步请求 Python 服务渲染 PDF。
  - **TODO 2**: `sign` 阶段未来将替换为获取第三方签章平台的跳转链接或签署 Token。
  - **TODO 3**: 签约成功后需调用 OSS Client 保存归档电子文件。
- **Rationale**：通过显式的 TODO，我们可以随时追踪需要重构的技术债。

## Risks / Trade-offs

**Risks:**
- 本次使用 JSON 替代 PDF 下发，前端只需调用 `/sign` 接口即可，未模拟出真实的签署意愿确认（如人脸+短信组合校验）和外部回调（Callback）异步场景。
  - **Mitigation**: 明确知会前端开发者，目前的同步 `/sign` 接口仅做流转控制；在后续开发正式版时，该部分会被拆分为两个阶段：前端拉起签章 SDK -> 后端接收外部异步回调并更新状态。
