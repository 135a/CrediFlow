## Why
在系统早期的前后端联调以及业务流转测试阶段，我们急需打通“借款申请 -> 合同签署 -> 放款受理”的全链路状态机。但如果在现阶段就接入真实的 PDF 渲染服务、第三方电子签章机构以及外部 OSS 文件存储，会带来极大的环境配置成本和网络依赖。为了保证能够快速开展核心业务流程测试，我们需要实现一个基于内存与 JSON 的“模拟签约”系统，以最低成本验证系统的状态流转机制。

## What Changes
- 在 `loan-contract-service` 提供基于纯内存（或单表简单记录）的**模拟合同生成能力**，合同内容表现为 JSON 结构，包含：合同编号、用户信息、借款信息、利率、状态。
- 实现一个极简的合同状态机：`INIT` (待签署) -> `SIGNED` (已签署)。
- 提供**模拟签署接口**：前端获取待签署的 JSON 合同后调用该接口确认，后端将状态置为 `SIGNED`，并以此作为触发后续放款流程的凭据。
- 提供**合同查询接口**：供前后端查阅合同明细与状态。
- 代码中将以明确的注释标明未来的 `TODO` 接入点（包含：Python 渲染 PDF、Go 网关电子签章、正式入库记录、OSS 上传以及预览/下载等）。

## Capabilities

### New Capabilities
- `mock-contract-signing`: 纯内存与 API 级的借款合同流转模拟能力，涵盖生成、查询、签署三大接口闭环。

### Modified Capabilities
- `loan-contract-repayment-plan`: 单笔借据生成规范可能涉及与模拟签署状态联动的细节更新（如签署完成后触发借据的流转）。

## Impact
- **代码影响**: 主要集中在 `loan-contract-service`，需要新增专门的 Controller 路由和 Service 模拟层来支撑联调。
- **依赖影响**: 完全**不依赖**外部第三方服务（解耦了 PDF、签章、OSS），降低了联调的部署要求。
- **后续规划**: 本次引入的设计应为未来的 `Facade` 或网关层代理保留出良好的扩展口，方便日后无缝切换为真实的签约引擎。
