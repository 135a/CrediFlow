## Context

CrediFlow 定位为「智能小额信贷」**后端综合系统**：逻辑接入层（多端只做角色与路由适配，不实现真实前端）→ **Apache APISIX** → **双端 BFF（管理端 / App 端各一可部署模块）** → **Spring Cloud Alibaba 领域微服务集群** → **Go 分布式任务调度** → **Python Data Agent** → 数据存储（MySQL / Redis / RocketMQ / 向量库）。同步调用以 **HTTP + OpenFeign** 为主，异步以 **RocketMQ** 解耦；目标交付形态以 **Docker Compose 单机编排** 为主，便于演示与简历级工程化说明。

本设计承接 `proposal.md` 中的能力拆分，把跨服务的关键技术选择与边界写清楚，为后续各 `specs/<capability>/spec.md` 提供一致约束。

## Goals / Non-Goals

**Goals:**

- 固化 **端到端分层架构** 与 **服务间通信策略**（网关 / Java / Go / Python / 消息 / 缓存 / DB）。
- 明确 **LLM 采用可插拔适配器**，并给出国内三家主流模型的 **选型对比与默认接入策略**（仍可通过配置切换实现方）。
- 明确向量检索使用 **Milvus**；部署默认 **Docker Compose**；合规文档采用 **通用原则 + 可审计清单**，不绑定单一司法辖区条文。
- 约定 **金融级横切能力**：幂等、审计日志、敏感数据保护、可观测性预留（与 `observability-platform` spec 对齐）。

**Non-Goals:**

- 不在本设计中单点落地「逐接口字段级」定义（归属各能力 spec）。
- 不承诺具体监管报送接口或牌照资质结论（仅工程与内控视角）。
- 本次变更范围仍以 **规格与设计文档** 为主，不要求在本 commit 内完成全部业务代码实现。

## Decisions

### D1. 总体拓扑与职责边界

- **APISIX**：唯一南北向入口；JWT 校验、限流、黑白名单、WAF 类防护、访问日志；**零业务规则**。
- **双端 BFF（两个独立 Maven/部署模块）**：
  - **`app-bff-service`（App 端）**：面向移动 App / 开放用户 API；路由前缀建议 **`/api/app/**`**；JWT 角色为 **终端用户**；可聚合多领域读模型、裁剪字段、统一错误码与版本协商；**禁止**实现后台特权操作。
  - **`admin-bff-service`（管理端）**：面向运营 / 风控 / 系统管理员；路由前缀建议 **`/api/admin/**`**；JWT 角色为 **管理员/运营**；可聚合报表、批量操作入口（仍调用领域服务，不在 BFF 写账本）；与 `system-service` 权限模型对齐。
  - 领域微服务（`user-service` 等）**不区分** App/Admin 两套实现，由网关 + BFF 按路由与令牌角色分流；BFF 与领域服务之间使用 **OpenFeign**，超时/熔断与 `X-Request-Id` 透传见集成 spec。
- **Java 微服务**：权威业务状态与事务边界；授信/申贷/合同/还款/贷后/资金流水/系统权限等按领域拆分（与 `proposal` 中各 `microservice-*` 一致）。
- **Go 调度服务**：时间驱动与高可靠批处理（代扣、巡检、罚息、提醒、异步风控投递、推送调度）；**不持有核心业务账本权威**，以调用 Java API / 发 MQ 为主。
- **Python Agent**：智能分析、RAG、NL2SQL/NL2API、报告与归因；通过 **HTTPS 内网调用** Java/只读数据接口；**不参与核心事务提交与最终授信裁决**，输出为建议/草稿/查询结果。

### D2. 服务发现、配置与韧性

- **Nacos**：注册发现 + 配置；Java 与（若纳入）Sidecar/网关上游发现策略在实现阶段与 Compose 网络 DNS 二选一或组合，本设计约束为：**对内服务名稳定、对网关暴露统一域名**。
- **Sentinel**：熔断限流以 **Java 服务 + APISIX 插件** 分层；限流维度默认：用户 ID、客户端 IP、关键写接口（放款、还款、合同签署）。
- **Spring Security + JWT**：业务服务间信任模型采用 **网关签发/校验 JWT**，内网服务可选择 **mTLS 或共享签名密钥** 二选一（实现阶段在 `tasks` 中择一固化）；本设计推荐：**网关强校验 + 内网最小信任**。

### D3. 数据与消息

- **MySQL**：主账本；**单库部署，不采用分库分表**（不引入 ShardingSphere 等分片中间件）；重要写路径要求 **事务 + 幂等键**（防重复放款/还款）。
- **Redis**：缓存、会话/令牌黑名单（若采用）、分布式锁；锁 key 与 TTL 规范在各业务 spec 展开。
- **RocketMQ**：领域事件与异步通知；**消费者幂等** 作为硬性要求。
- **Milvus**：金融知识库/政策片段/内部规程等 **向量化存储与检索**；与 RAG 管道同 VPC/Compose 网络互通。

### D4. LLM：可插拔适配器与国内三家模型

**设计原则**：Agent 侧仅依赖稳定抽象（例如 `LLMClient.generate / stream`、`embeddings`），具体厂商通过 **适配器 + 配置** 注入；密钥来自 **环境变量/密钥管理**，禁止写入仓库。

**选型对比（用户确认维度）**

| 选型 | 推荐理由 | 适用场景 |
| --- | --- | --- |
| **通义千问（阿里云）** | 国内生态成熟，企业级服务稳定，API 成本中等，支持自定义微调 | 后端服务、企业级应用、已有阿里云基础的项目 |
| **智谱清言（智谱 AI）** | 多模态能力较强，对话逻辑严谨，金融文本场景适配较好 | 信贷风控辅助、文本分析、需要高可靠性对话的场景 |
| **文心一言（百度）** | 国内市场占有率高，企业服务方案完善，合规流程成熟 | 对合规、审计流程要求严格的金融项目 |

**默认策略（可改）**：若基础设施已阿里云化，**默认适配器优先接「通义千问」**；若强调贷后文本与风控对话链路的稳定性，可在配置中切换 **智谱**；若组织更强调企业合规流程与审计材料完备度，可切换 **文心一言**。三套适配器 **并存于代码结构**，运行时 **单活**（一个 `ACTIVE_PROVIDER`），避免混用导致计费与审计不可追踪。

### D5. Agent 技术栈与 Milvus

- **编排**：LangGraph（多步工具调用、审批人类节点预留）。
- **RAG**：文档切片 → embedding → **Milvus** 检索 → 重排（可选）→ LLM 生成；引用片段需 **回传 source id** 便于审计。
- **NL2SQL**：仅允许连接 **只读从库/只读账号/行级权限视图**；必须 **SQL 静态分析与白名单表**；结果以结构化 JSON 返回，不直接对生产写库。
- **NL2API**：映射到 **已发布、已鉴权** 的后台 API 清单（白名单 + 参数 schema 校验），禁止开放式任意 HTTP。

### D6. 部署：Docker Compose

- **目标**：单机 `docker compose up` 拉起 **依赖中间件 + 网关 + 各语言服务** 的最小可演示拓扑（具体 `compose.yaml` 在实现阶段落地）。
- **网络**：划分 `edge`（APISIX 对外）与 `internal`（微服务/DB/MQ/向量库仅内网）。
- **数据卷**：MySQL、Milvus、RocketMQ 持久化卷策略在实现任务中定义；开发环境允许简化副本数。

### D7. 合规与审计（通用原则 + 清单）

- **原则**：最小必要、目的限制、分级授权、可追溯、可恢复演练（备份策略在运维 spec/任务中展开）。
- **可审计清单（设计层约束）**：身份鉴别、敏感操作双人复核预留（可选开关）、全链路 request id、网关访问日志、业务审计表、合同与资金操作留痕、Agent 输出保留引用来源与模型版本/提示词版本号（**不含**密钥与完整 PII）。

## Risks / Trade-offs

- **[Risk] LLM 幻觉导致错误建议** → **缓解**：授信/放款等终局动作必须 Java 规则引擎 + 人工/流程节点；Agent 输出仅「辅助」；关键字段二次校验。
- **[Risk] NL2SQL 数据泄露** → **缓解**：只读隔离、表白名单、字段掩码、查询结果行数上限、审计日志。
- **[Risk] 单机 Compose 与生产高可用差距** → **缓解**：设计明确区分「演示拓扑」与「生产 K8s 外扩」；HPA/多副本不在本阶段非目标内强制。
- **[Risk] 多模型适配维护成本** → **缓解**：适配器接口稳定、集成测试用固定 mock；各厂商差异收敛在适配器层。
- **[Trade-off] 网关 vs 服务双重限流** → 可能重复；通过「网关防刷、服务防雪崩」分层阈值避免互相干扰。

## Migration Plan

1. **先中间件与观测基座**：MySQL/Redis/RocketMQ/Nacos/Milvus/Prometheus（可选）在 Compose 中就绪。
2. **再网关**：APISIX 路由、JWT、基础限流与日志。
3. **Java 核心域**：用户 → 授信 → 申贷 → 合同 → 还款最小闭环接口与审计字段。
4. **Go 调度**：只读/回调型任务先接入，避免与未就绪写路径耦合。
5. **Python Agent**：适配器接所选 LLM，RAG 最小知识库导入，NL2SQL/NL2API 白名单最小集。

**回滚**：Compose 版本化；数据库迁移脚本可逆策略（或快照恢复）在 `tasks` 细化；网关路由支持快速切回维护页。

## Open Questions

- **内网互信模型**：Java 集群是否统一采用 **服务间 JWT** 还是 **mTLS**（影响开发与运维复杂度）。
- **人脸/短信等第三方**：具体供应商与合规协议文本归属业务采购与法务，本设计仅保留接口与数据留存策略占位。
